# 命書 80% 資料庫 + 20% GPT-5.2 潤飾：可行性與待補資料規劃

## 一、星曜介紹來源釐清

| 來源 | 提供內容 | 命書／內容用途 |
|------|----------|----------------|
| **iztro** | 命盤**結構**：星名（soul/body）、宮位索引、地支、四化等。**不提供**星曜的「文字介紹」或「在該宮位的表現」。 | 算盤、取星名與宮位對應，供 content key 查表。 |
| **你們的 content（D1／靜態 JSON）** | **stars**（`star.紫微`）：該星的「核心本質、成熟/失衡、課題」等一段說明。<br>**starPalaces**（`starPalace.紫微_命宮`）：該星在該宮位的表現（小句或短段）。 | 命書組裝時查 `content.stars`、`content.starPalaces`，即「星曜介紹」與「星曜×宮位」的**唯一文案來源**。 |

結論：**星曜的「介紹」與「在不同宮位表現」都來自你們的資料庫／靜態 content，不是 iztro。** iztro 只給「這張盤命宮是哪顆星、官祿宮是哪顆星」等結構。

## 二、待補資料規劃校對與修正

以下依你們的類別逐項對齊現有結構，並標註**筆數／字數是否合理**、**建議修正**。

### 2.1 22 章 × 4 段 → 88 段，約 4–6 萬字

| 項目 | 說明 |
|------|------|
| **規劃** | 正確。22 section × 4 欄（structure_analysis, behavior_pattern, blind_spots, strategic_advice）= 88 段。 |
| **字數** | 4–6 萬字合理（每段約 450–680 字）。若希望 80% 由 DB 輸出，這 88 段即「章節骨架」，可寫成帶占位符的模板（如 `{宮位}」「{十神}」「{五行}」「{主星列表}」），組裝時替換。 |
| **key 建議** | `lifebookSection.{s00～s21}` 一筆 JSON 含四欄，或拆成 `lifebookSection.{key}.structure_analysis` 等四種 key。 |

### 2.2 星曜 base 模板 → 14 × 2 段 = 28，5k–8k 字

| 項目 | 說明 |
|------|------|
| **規劃** | 14 = 14 顆主星（與 star-registry 一致）。現有 `content.stars` 是**每星一段**，你們要擴成**每星 2 段**（例如「核心本質」+「實務提醒」），故 14×2=28 段合理。 |
| **字數** | 5k–8k 字 ÷ 28 ≈ 180–285 字/段，合理。 |
| **與現有對應** | 現有 `stars["紫微"]` 為單一段落；可改為 JSON 存兩段，例如 `{ "base": "…", "practice": "…" }`，或拆成 `starBase.紫微`、`starPractice.紫微`。 |
| **輔星** | 若只做主星 14，不需改筆數。若未來要「星曜 base」含常用輔星，再擴充（見下方星曜×宮位）。 |

### 2.3 星曜 × 宮位小句 → 14 × 12 = 168，約 5k 字

| 項目 | 說明 |
|------|------|
| **規劃** | 即現有 **starPalaces**（`starPalace.{星名}_{宮名}`）。14 主星 × 12 宮 = 168 筆，與現有 content-zh-TW 結構一致。 |
| **字數** | 5k 字 ÷ 168 ≈ 30 字/句，屬「小句」。你們現有 starPalaces 多為 20–40 字/條，若統一壓在約 30 字可達 5k。 |
| **輔星 × 宮位** | 若要做「完整星曜」（含常用輔星）：doc 建議約 27 輔星 × 12 宮 = **324 筆**，約再 5k 字（小句）。**合計 14×12 + 27×12 = 168 + 324 = 492 筆，約 1 萬字。** 你們表上僅列 168 筆，代表先做**主星**即可；輔星可列為 Phase 2。 |

### 2.4 十神 × 宮位 snippet → 50 筆，5k–8k 字

| 項目 | 說明 |
|------|------|
| **規劃** | 完整格為 **10 十神 × 12 宮 = 120 筆**（tenGodPalace.{十神}_{宮名}）。你們寫 50 筆，可能為「先做 50 個常用組合」。 |
| **建議** | 若 80% 內容來自 DB，建議**直接規劃 120 筆**，避免命盤出現某十神×宮位時查無內容。字數 5k–8k 對 120 筆 ≈ 40–65 字/筆，合理。 |
| **修正建議** | 類別改為「十神×宮位 snippet」**120 筆**，約 5k–8k 字；若分階段則 Phase1 先 50 筆、Phase2 補滿 120。 |

### 2.5 高壓 pattern（最重要）→ 30～50 筆，6k–10k 字

| 項目 | 說明 |
|------|------|
| **規劃** | 屬**新增類別**，對應「化忌疊加、超級地雷區、雙忌、雙祿、吉凶並見」等情境的**說明＋建議**文案。 |
| **用途** | 組裝時依 overlapAnalysis（criticalRisks、volatileAmbivalences、maxOpportunities）或 traffic_signals 選用對應 snippet，拼進該章或該宮的 structure_analysis／strategic_advice。 |
| **key 建議** | 例如 `highPressure.雙忌`、`highPressure.超級地雷區`、`highPressure.吉凶並見`、或依宮位 `highPressure.雙忌_財帛` 等；可先 30～50 條通用＋少量宮位專用。 |
| **筆數與字數** | 30～50 筆、6k–10k 字合理（每條約 200 字上下）。 |

### 2.6 五行弱項 → 5 × 2 段 = 10，約 1k 字

| 項目 | 說明 |
|------|------|
| **規劃** | 即現有 **wuxing** 弱項建議的擴充：5 行 × 2 段（例如「特徵」+「補運建議」）= 10 段。 |
| **與現有對應** | 目前 lifeBookTemplates 的 WUXING_WEAK_PHRASES 為 5 筆單段；改由 D1 提供時可改為 5 筆 × 2 段，或 5 筆 JSON `{ "weak": "…", "advice": "…" }`。 |
| **字數** | 1k 字合理。 |

### 2.7 命主身主模板 → 10～14 筆，約 2k 字

| 項目 | 說明 |
|------|------|
| **規劃** | 命主／身主只會落在**特定星**（依命宮地支、年支對照表），約 12 顆星會輪流當命主、身主。10～14 筆即這些星的「當命主時／當身主時」的說明。 |
| **與現有對應** | 現有 `content.stars` 已可當「該星」的通用說明；若要做「命主專用／身主專用」文案，可新增 `starAsMingzhu.紫微`、`starAsShenzhu.紫微` 等，或沿用 stars 並在組裝時標註「命主星／身主星」。 |
| **筆數** | 10～14 筆、2k 字合理。 |

## 三、修正後總表與最終資料量級（含星曜）

| 類別 | 建議筆數 | 文字量 | 備註 |

| 星曜 base 模板 | 14 × 2 = 28 | 5k–8k 字 | 主星兩段式，對應 content.stars 擴充 |
| 星曜 × 宮位小句 | 14 × 12 = 168 | 約 5k 字 | 即 starPalaces 主星，已部分存在 |
| 星曜（輔星）× 宮位 | 27 × 12 = 324（Phase2） | 約 5k 字 | 可選，總計 492 筆星×宮 |
| 十神 × 宮位 snippet | **120**（建議補滿） | 5k–8k 字 | 10×12；若分階段則先 50 再 70 |
| 高壓 pattern | 30～50 | 6k–10k 字 | 雙忌／雷區／吉凶並見等 |

| **含輔星×宮位** | **+324 = 約 782～826 筆** | **+約 5k 字** | Phase2 可做 |

### 3.2 與「小句 300–500 句」的對應

- 「小句」可定義為：星曜×宮位 168（＋輔星 324）、十神×宮位 120、高壓 pattern 部分短句，合計約 300–500 句合理。
- 若只算「一句話級」：168 + 120 = 288，再加高壓 30–50 條中的短句，即落在 300–500 區間。

### 3.3 最終資料量級（含星曜）

- **Phase1（主星 only）**：約 **460～500 筆** snippet／段，總字數 **8～15 萬字**。
- **Phase2（含輔星×宮位）**：再 **+324 筆**、約 **+0.5 萬字**，總計約 **780～830 筆、8.5～15.5 萬字**。

你們原表「200～330 snippet」若指「**大類條目數**」（例如 22 章、14 星、1 組十神宮位等）則偏少；若指「**可獨立查表的 key 數量**」，則 88+28+168+120+40+10+12 ≈ **466** 更接近實際，建議以 **450～500 筆** 為 Phase1 目標，**含星曜後 500 筆、8～15 萬字** 為佳。

|------|------|

## 五、操作建議（80% DB + 20% GPT-5.2 潤飾）

### 5.1 組裝流程（建議）

1. **依 section_key 與命盤維度查表**  
   - 讀取 lifebookSection、starPalaces（該宮主星／輔星）、tenGodPalaces、wuxingPalaces、stars（命主身主）、wuxing 弱項、highPressure（依 overlap 結果選用）。
2. **拼出「粗稿」四欄**  
   - 替換占位符（宮位、十神、五行、主星列表等），產出每章 4 段正文（80% 內容）。
3. **呼叫 GPT-5.2 僅做潤飾**  
   - 輸入：該章四段粗稿 + 簡短指示（例如：「僅潤飾語氣與銜接，不改動事實與建議要點，輸出同一四欄 JSON」）。  
   - 參數：temperature=0（或該模型最低）、max_tokens 適度、system 簡短固定（「你是潤稿員，只做語氣與銜接，不新增內容」）。
4. **輸出**  
   - 與現有命書 JSON 格式一致，便於現有 Viewer／儲存流程使用。

### 5.2 待補資料優先順序（建議）

1. **第一批**：22 章×4 段（88 段）、星曜 base 14×2（28 段）、星曜×宮位 168（主星）、十神×宮位 120（或先 50）。  
   → 可先上線「主星 + 十神 + 五行」的 80% 組裝。
2. **第二批**：高壓 pattern 30～50、五行弱項 10、命主身主 10～14。  
   → 補齊雷區／吉凶並見與命主身主體驗。
3. **第三批（可選）**：輔星×宮位 324。  
   → 星曜覆蓋完整，總資料量級約 780+ 筆、8.5–15.5 萬字。

### 5.3 Worker／API 建議

- 新增 mode 或 endpoint（例如 `?mode=hybrid` 或 `/api/life-book/assemble-and-polish`）：  
  - 先依上述邏輯做 DB 組裝，再呼叫 GPT-5.2 僅做潤飾。  
- 保留現有「全 GPT 生成」為 fallback 或 A/B 測試。  
- 潤飾時傳入的 token 以「該章四段」為主，可大幅壓低 token 消耗與延遲。

## 六、總結

- **星曜介紹**：來自你們的 **content.stars + content.starPalaces**，不是 iztro；iztro 只提供命盤結構。
- **待補規劃**：整體正確；建議將十神×宮位改為 **120 筆**（或 50+70 分階段），其餘筆數與字數維持。
- **最終資料量級（含星曜）**：Phase1 約 **450～500 筆、8～15 萬字**；含輔星×宮位則約 **780～830 筆、8.5～15.5 萬字**。
- **80/20 + GPT-5.2**：可行；實作上先 DB 組裝 80%，再單次呼叫 5.2 做潤飾（無 temperature、自由度受控），品質與成本較易兼顧。
