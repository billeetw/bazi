## 二、技術債與優化建議

### 🔴 高優先級

#### 1. **API 層重複程式碼**

**現況**：`parseBasicAuth`、`jsonResponse`、`unauthorized` 在 20+ 個 `functions/api/*.js` 中重複實作。

**建議**：抽出共用模組 `functions/_shared/admin-auth.js`：

```javascript
// functions/_shared/admin-auth.js
export function parseBasicAuth(request) { ... }
export function jsonResponse(data, status) { ... }
export function unauthorized() { ... }
export function requireAdmin(request, env) { ... }
```

各 API 改為 `import { parseBasicAuth, jsonResponse } from '../_shared/admin-auth.js'`。

---

#### 2. **divination.html 內嵌邏輯過重**

**現況**：約 1000 行 inline script，含資料載入、UI 渲染、事件綁定、歷史紀錄等，難以維護與測試。

**建議**：
- 抽出 `js/divination-app.js`（或納入 Vite 建置）
- 將 `showResult`、`renderLines`、`loadDivinationStats` 等拆成可測試函式
- 與主站共用 `auth-service`、`getAuthHeaders` 等，避免重複

---

#### 3. **獨立頁面未使用 bundle**

**現況**：`divination.html`、`consultation.html` 直接載入多個 script，無 tree-shaking、無 hash 檔名（不利快取）。

**建議**：
- 新增 `vite.divination.config.ts`，產出 `dist/divination.js`
- 或將 divination 納入主 bundle 並用動態 import 按需載入（若 SEO 允許）

---

#### 4. **API 路由分散：Pages vs Worker**

**現況**：部分 API 在 Pages Functions 執行（如 divination、consultation、auth），部分 proxy 到 Worker（compute、content、log-usage）。邏輯分散，除錯需兩邊看。

**建議**：
- 短期：在 `docs/` 維護「API 路由對照表」，標註執行位置
- 長期：評估是否將所有 API 統一到 Worker，Pages 只做 proxy，簡化部署與權限管理

---

### 🟡 中優先級

#### 5. **專家後台 (expert-admin.html) 體積大**

**現況**：單一 HTML 約 2200+ 行，含大量 inline script，與主站邏輯部分重複。

**建議**：
- 將「占卦使用與建議」「推算時辰結果」等區塊拆成獨立模組
- 共用 `authHeader()`、`auth` 等登入邏輯

---

#### 6. **D1 查詢未統一錯誤處理**

**現況**：各 API 對 `env.CONSULT_DB` 的檢查與錯誤回傳方式不一致。

**建議**：建立 `getDb(env)` 輔助函式，統一回傳 `{ db, error }`，呼叫端一致處理。

---

#### 7. **Migration 與部署未自動化**

**現況**：`deploy.sh` 不執行 migration，需手動 `npx wrangler d1 migrations apply`。

**建議**：在 `deploy.sh` 中加入 migration 步驟（或至少在 CI 中執行），並在文件中註明。

---

### 🟢 低優先級

#### 8. **測試覆蓋率**

**現況**：有 `test:e2e`，但核心 calc 邏輯（如 `dayanDivination`、`baziCore`）缺乏單元測試。

**建議**：為關鍵演算法補上 Jest/Vitest 單元測試，避免重構時回歸。

---

#### 9. **型別與文件**

**現況**：`js/` 多為純 JS，`worker/` 有 TypeScript，型別不統一。

**建議**：逐步為共用介面（如 API 回應格式）加入 JSDoc 或 `.d.ts`，方便 IDE 與重構。

---

#### 10. **多餘與測試用檔案**

**現況**：`test-*.html`、`admin-copy.html`、`index3.html`、`ui.js.bak` 等可能為歷史遺留。

**建議**：確認無用後移至 `archive/` 或刪除，減少混淆。

---

## 三、優化優先順序建議

| 優先級 | 項目 | 預估工時 | 效益 |
|--------|------|----------|------|
| 1 | API 共用 auth 模組 | 0.5 天 | 減少重複、易維護 |
| 2 | divination 邏輯抽出 | 1–2 天 | 可測試、易擴充 |
| 3 | divination 納入 Vite bundle | 0.5 天 | 快取、tree-shaking |
| 4 | 部署流程加入 migration | 0.5 天 | 降低人為疏漏 |
| 5 | API 路由對照文件 | 0.5 天 | 除錯與 onboarding |
| 6 | 清理測試/備份檔案 | 0.5 天 | 程式庫更乾淨 |

---

## 四、小結

目前架構能支撐公測與日常營運，主要技術債集中在：

1. **重複程式碼**：API 層 auth、divination 內嵌邏輯  
2. **建置與部署**：獨立頁未 bundle、migration 未自動化  
3. **可維護性**：大型 inline script、分散的 API 路由  

建議先處理「API 共用 auth」與「divination 邏輯抽出」，再視需求逐步進行其餘優化。
