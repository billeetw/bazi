# 占卦登入儲存紀錄 — 可行性評估與延伸效益

> 評估「登入用戶可儲存占卦紀錄」的實作難度、整體規劃與可延伸服務效益。

---

## 一、現況盤點

| 項目 | 狀態 | 說明 |
|------|------|------|
| `divination_logs` 表 | ✅ 已有 | 含 user_id、question、mood、卦象、lines_json 等 |
| `POST /api/divination` | ✅ 已支援 user_id | 若有 JWT，會自動寫入 user_id |
| divination.html 傳 JWT | ❌ 未實作 | 目前未帶 Authorization，一律匿名儲存 |
| `GET /api/me/divinations` | ❌ 無 | 無法查詢用戶的占卦紀錄 |
| 占卦頁登入 UI | ❌ 無 | 占卦頁為獨立頁，未載入 AuthService |
| 紀錄列表 / 查看歷史 | ❌ 無 | 無 UI 可瀏覽過往占卦 |

**結論**：後端已具備「登入即儲存」的基礎，缺的是前端串接與查詢 API。

---

## 二、可行性評估

### 2.1 技術難度：低～中

| 項目 | 難度 | 說明 |
|------|------|------|
| 占卦頁帶 JWT 儲存 | 低 | 載入 auth-service，fetch 時加 getAuthHeaders() |
| 占卦頁登入入口 | 低 | 加「登入可儲存紀錄」提示 + 連結至首頁登入 |
| `GET /api/me/divinations` | 低 | 仿照 /api/me/charts，查 divination_logs WHERE user_id |
| 紀錄列表 UI | 中 | 新增「我的占卦紀錄」區塊或獨立頁 |
| 查看歷史結果 | 中 | 用 lines_json 還原卦象，呼叫現有 showResult 邏輯 |

### 2.2 實作範圍建議

**階段一（MVP，約 1–2 天）**
1. divination.html 載入 auth-service.js
2. POST /api/divination 時帶 Authorization（若已登入）
3. 占卦頁加「登入可儲存紀錄」提示，連結至 index.html 登入
4. 成功儲存後顯示「已儲存至你的紀錄」（僅登入時）

**階段二（紀錄列表，約 2–3 天）**
1. 新增 `GET /api/me/divinations`：回傳用戶的占卦列表（id, created_at, question, primary_index, 卦名）
2. 占卦頁或首頁加「我的占卦紀錄」區塊（僅登入時顯示）
3. 點選某筆紀錄 → 還原並顯示該次結果（重用現有 renderLines、卦辭等）

**階段三（進階，可選）**
1. 紀錄上限（如每人最多 20 筆，超過刪最舊）
2. 刪除單筆紀錄
3. 紀錄與命盤關聯（若用戶有命盤，可做卦象 × 八字交叉分析）

## 三、整體規劃

### 3.1 資料流

```
用戶占卦完成
    ↓
POST /api/divination（帶 JWT 若已登入）
    ↓
寫入 divination_logs（user_id 有值 = 已儲存）
    ↓
GET /api/me/divinations（登入後可查）
    ↓
「我的占卦紀錄」列表 → 點選 → 還原結果頁
```

### 3.2 與現有服務的關係

| 現有服務 | 占卦儲存後的關聯 |
|----------|------------------|
| 命盤（user_charts） | 同一 user_id；未來可做「卦象 × 八字」交叉分析 |
| 諮詢（consultations） | 占卦紀錄可作為諮詢前的「問題脈絡」參考 |
| 命書（life_books） | 可推薦「用此卦象深化諮詢」→ 命書 |
| 推算時辰（estimate_hour_logs） | 同為 /api/me/*，共用 JWT 與登入流程 |

## 四、可延伸服務效益

### 4.1 直接效益

| 效益 | 說明 |
|------|------|
| 用戶留存 | 有紀錄可回顧，提高回訪意願 |
| 深度互動 | 多筆紀錄可觀察「同一問題的卦象變化」 |
| 登入誘因 | 「登入可儲存」促使匿名用戶註冊 |
| 數據價值 | 可分析熱門卦象、常見問題類型、情境偏好 |

### 4.2 轉換與營收延伸

| 延伸服務 | 說明 |
|----------|------|
| 占卦 → 諮詢 | 低分月份、動爻轉折 → CTA「預約 1:1 諮詢」 |
| 占卦 → 命書 | 「用此卦象深化」→ 命書付費流程 |
| 占卦 → 命盤 | 已有「查看命盤」CTA；儲存紀錄可強化「回顧卦象 + 對照命盤」 |
| 動爻月提醒 | 依儲存紀錄的 timePeriod，推算動爻月 → 推播/Email 提醒（已有 divinationReminders 邏輯） |

### 4.3 內容與體驗延伸

| 延伸 | 說明 |
|------|------|
| 卦象趨勢 | 同一問題多次占卦 → 簡易「卦象變化」時間軸 |
| 文章推薦 | 依歷史卦象推薦相關文章（如困卦 → 困卦解讀文） |
| 年度回顧 | 年末「你的 2026 占卦回顧」：幾卦、熱門主題、動爻月提醒 |
| 分享優化 | 儲存紀錄可生成「專屬分享卡」（含歷史卦象摘要） |

## 五、實作優先順序建議

| 優先 | 項目 | 預估工時 | 效益 |
|------|------|----------|------|
| 1 | 占卦頁帶 JWT 儲存 + 登入提示 | 0.5 天 | 立即可儲存，無需新 API |
| 2 | GET /api/me/divinations | 0.5 天 | 可查紀錄 |
| 3 | 我的占卦紀錄 UI（列表 + 查看） | 1–2 天 | 完整體驗 |
| 4 | 紀錄上限、刪除 | 0.5 天 | 避免無限增長 |
| 5 | 占卦 × 命盤交叉分析 | 2+ 天 | 進階差異化 |

---

## 六、注意事項

1. **隱私**：占卦問題可能涉及敏感內容，需在隱私政策說明儲存目的與保留期限。
2. **紀錄上限**：建議每人 20–50 筆，避免 D1 寫入過多。
3. **匿名仍可用**：未登入時照常占卦，僅不儲存；不強制登入。
4. **divination.html 獨立性**：占卦頁目前不載入主站 app.js，需單獨載入 auth-service.js 或抽一組最小 auth 依賴。
