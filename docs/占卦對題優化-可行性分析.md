# 占卦對題優化 — 可行性分析

## 一、目前卡住的 3 個點（建議對應）

| 卡點 | 建議解法 | 可行性 |
|------|----------|--------|
| 1. 「解讀情境」同時列 4 個，與「你問的是【事業】」重複，造成困惑 | 只顯示「本次情境：事業」，其他放「切換解讀」區 | ✅ 高 |
| 2. 問題內容沒有進入回覆 → 仍被評為不相關 | 開頭固定加「你問的具體問題是：『＿＿＿＿＿』」 | ✅ 高 |
| 3. 資訊密度太高，重點太晚出現 | 三層閱讀：第一屏摘要 → 第二屏因果 → 第三屏易經學習（折疊） | ⚠️ 中（需你提供 1 項） |

---

## 二、各項建議可行性細分

### 1) 解讀情境 UI 調整 — ✅ 可行

**現況**：`contextToggleWrap` 顯示 4 個按鈕（事業/財運/感情/無特定），與「你問的是【事業】」並列。

**改法**：
- 第一屏只顯示：`本次情境：事業`（由 analysis.category 決定）
- 4 個按鈕移到「切換解讀」折疊區，或改為小字「想看財運視角？點此切換」

**工量**：小，約 30 分鐘。

---

### 2) 問題內容進入回覆 — ✅ 可行

**現況**：`questionRestatementText` 只顯示「你問的是【事業】的整體趨勢」，沒有原問題。

**改法**：
```
你問的具體問題是：「今年事業整體趨勢如何？」
類型判定：事業｜趨勢
一句話結論：吉，但 6 月需守；12 月慎收尾
```

**工量**：小，約 15 分鐘。我們已有 `question` 變數，直接顯示即可。

---

### 3) 三層閱讀 — ⚠️ 需你提供 1 項

**A. 對題摘要（第一屏）** — 部分可行

| 區塊 | 現有資料 | 缺口 |
|------|----------|------|
| 你問的是（原問題） | ✅ question | 無 |
| 類型判定 | ✅ category, intent | 無 |
| 一句話結論 | ✅ totalTrend（吉/凶/平） | 需規則：如何組合「但 X 月需守」「Y 月慎收尾」 |
| 三步行動 | ⚠️ hexagram-actions 有 solutionKeywords、learningTasks | **需你決定**：用現有文案，還是要「時間窗 + 行動」格式（如「2–4 月：把資源轉制度」） |
| 時間窗（高點/轉折/風險） | ✅ computeMonthlyScores、changingLines | 需依 timePeriod 轉換標籤（已有 formatTimeLabel） |

**B. 為什麼（第二屏）** — ✅ 可行

本卦/互卦/之卦解釋已有，只需調整排版與折疊邏輯。

**C. 易經學習卡（第三屏，折疊）** — ✅ 可行

卦辭、彖象、當位應與、五行、方位等已有，改為預設折疊即可。

**D. 回饋可驗證點** — ✅ 可行

在回饋區加「本次可驗證點」+ 選項（有/沒有/部分），需從卦象推出一句可驗證描述（可用動爻或之卦）。

---

## 三、需要你提供的 3 項（關鍵）

### 1. 三步行動的來源與格式

**選項 A**：直接用 `hexagram-actions.json` 的 `solutionKeywords`、`learningTasks`  
- 優點：已有 64 卦完整資料，可立刻上線  
- 缺點：沒有「2–4 月」「6–8 月」這種時間窗，較像通用建議  

**選項 B**：建立「本卦 + 互卦 + 之卦」→ 行動模板對照表  
- 範例：泰 + 歸妹 + 復 → 三條時間窗行動  
- 優點：對題感強、像在回答具體問題  
- 缺點：64×64×64 組合太多，需你提供「高頻組合」或規則（例如：本卦決定主軸、互卦決定過程、之卦決定收尾）  

**選項 C**：混合 — 用 hexagram-actions 當基底，依 intent 微調  
- 例如 strategy → 強調 learningTasks；timing → 強調「高點推進、轉折保守」  

**請你決定**：要 A、B、C 哪一種？若選 B，能否提供 10–20 組「本卦+互卦+之卦 → 三條行動」的範例？

---

### 2. 一句話結論的組合規則

範例：`吉，但 6 月需守；12 月慎收尾`

**現有資料**：
- `totalTrend`：吉 / 小吉 / 平 / 小凶 / 凶  
- `changingLines`：動爻索引 → 對應月份（依 timePeriod）  
- `riskMonths`：分數 ≤ 0 的月份  

**需你決定**：
- 「需守」對應動爻月，還是低分月？或兩者都要？  
- 「慎收尾」只給最低分的那個月，還是所有風險月都列？  
- 若無動爻、無風險月，結論是否只顯示 totalTrend（如「吉」）？  

**暫定規則**（可再調整）：
```
結論 = totalTrend
     + (有動爻月 ? "，但 X、Y 月需留意轉折" : "")
     + (有風險月 ? "；Z 月收尾宜慎" : "")
```

---

### 3. 專案結構：維持 js/ 還是遷到 src/？

**現況**：`js/calc/divinationQuestionAnalyzer.ts`、`js/divination-app.js`，無 `src/domain/`。

**選項 A**：維持 `js/`，在 `js/calc/` 新增 `buildCoreSummary.ts`  
- 與現有 divination 模組一致，改動小  

**選項 B**：依建議建立 `src/domain/question/`、`src/domain/reading/`、`src/ui/`  
- 結構較清楚，但需調整 Vite 與 divination 入口  

**建議**：先採 A，等第一屏穩定後再考慮重構。
