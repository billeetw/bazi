# 逐宮讀者敘事：確定性內容框架（無 LLM 渲染、成本可控）

> **目標**：在**既有管線**（`NormalizedChart` → `PalaceNarrativeBuilder` → `renderPalaceNarrativeSample`）內，用**規則 + 語料**盡量逼近「完美」，但**組合數可預估、維護不爆炸**，讓產品能**低成本服務最多人**。  
> **原則**：**不用 LLM 做章節正文渲染**；若日後只在**離線編輯／輔助寫句庫**用模型，與本框架不衝突。

**怎麼一段一段補**：請用 **[補資料順序（橫切 12 宮 vs 直切單宮）](./lifebook-reader-narrative-supplement-sequence.md)** — 可選「一次一項目填滿 12 宮」或「依 rollout 一宮到底」。

**（可選）讀起來更像人、但不加段落**：見 **[Behavior Axis、語氣層與循環鏈（BM3）](./lifebook-behavior-axis-tone-and-loop.md)** — Behavior Axis→語氣偏移、星曜×軸衝突、H3×H4×Loop；**待算法提供後**再評估 PoC。

---

## 一、什麼叫「完美」在此專案裡可落地

| 層次 | 定義 | 是否必達 |
|------|------|----------|
| **結構一致** | 每宮讀者文都有固定欄位（斷語→…→牽動等），同一盤可重現 | ✅ 必達（已由架構保證） |
| **語意同源** | 星曜／四化敘事與模組二、`narrativeFacade` 等同源矩陣，避免兩套矛盾 | ✅ 必達（持續對齊） |
| **多數盤讀起來「像人話」** | 常見主星、雙主星、權重段落不空洞 | ✅ 產品目標 |
| **每一盤都獨一無二像諮詢** | 純規則無法保證 | ❌ 不列為本框架的「完成條件」 |

**結論**：「完美」在此 = **在約束內的可驗收品質**，不是無限個案客製。

## 二、三層內容模型（已存在 + 你補齊的位置）

```
┌─────────────────────────────────────────────────────────────┐
│ L1 通用底層（全宮共用，規模固定）                              │
│  • PALACE_SEGMENT_SCHEMA（宮主題軸）                           │
│  • starSemanticDictionary（星曜 plain/risk/core…）             │
│  • 權重分段（core/decision/phenomenon/pitfall）               │
│  • buildSchemaDrivenPalacePremium（batch-2 斷語／問／鏡像／轉念） │
└─────────────────────────────────────────────────────────────┘
                              ↓ 不足時
┌─────────────────────────────────────────────────────────────┐
│ L2 宮別專屬語料（有上限地加）                                  │
│  • wealthPalaceProfiles／careerProfiles／tianZhai…            │
│  • mingReaderPremium 規則表（命宮條件包）                      │
│  • palaceReaderPremium：財／官／夫 batch-1、福德等雙星專包     │
└─────────────────────────────────────────────────────────────┘
                              ↓ 再不足時
┌─────────────────────────────────────────────────────────────┐
│ L3 窄規則命中包（最少筆數、最高 ROI）                          │
│  • 條件：主星 + 宮 +（可選）生年四化／亮度閾值                  │
│  • 一條規則 = 一套 headline / questions / mirror / turn      │
│  • 寧可少而準，不可多而碎                                     │
└─────────────────────────────────────────────────────────────┘
```

**成本控制關鍵**：runtime **只做查表 + 字串組裝**，複雜度 O(規則數)，規則數由你**顯式預算**。

## 三、規模控制：避免「組合爆炸」的硬規則

1. **優先補 L1**：字典缺句、某星在某宮 risk 空 —— 影響面最大，改一行全宮受益。
2. **L2 按「宮」配預算**：例如每宮最多 **N 條**專屬 profile（你自訂 N，建議先小後大）。
3. **L3 規則用「命中序」**：同一盤只取**第一個命中**的 premium 包（與命宮 `MING_PREMIUM_RULES` 同思路），避免多規則疊加。
4. **雙主星專包**：只對**資料或驗收證明高頻**的組合開專包（例：福德・天同巨門）；其餘走 L1/L2。
5. **Fallback 永遠存在**：無命中時回到 schema 泛用句 —— **寧可略平淡，不可錯或無窮維護**。

## 六、「完成」的可操作定義（建議你採用其一）

- **版本完成 v1.0**：12 宮皆可生成、清單驗收通過、無阻斷錯誤；L3 規則條數 ≤ 你設定的預算 K。  
- **版本完成 v1.1**：在 v1.0 上，依 **rollout 順序**（見 `lifebook-per-palace-reader-narrative-rollout-checklist.md`）補滿前四宮 + 指定雙星包。  
- **持續迭代**：每季只加 **固定條數**規則或 **固定工時**補字典，避免無上限膨脹。

---

## 七、為什麼這樣仍能「造福世界」

- **邊際成本極低**：每多一個使用者，不增加推理費；只有伺服器與儲存。  
- **品質可測**：同一 `chart` 輸出可 snapshot 測試，回歸可自動化。  
- **透明可維護**：規則與語料在 repo 內，不依黑盒模型。

---

## 八、與 LLM 的邊界（符合你的原則）

| 允許 | 不建議作為「正式渲染」 |
|------|-------------------------|
| 離線輔助：幫你起草句庫再人工審 | 線上對每 user 即時生成整章 |
| 內部工具：從真盤 log 聚類缺句 | 將 LLM 輸出直接當最終命書正文 |

若堅持 **全程無 LLM**，仍可依本框架只用手寫 + 試盤迭代完成 v1.x。
