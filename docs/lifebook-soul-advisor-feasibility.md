# 命書 Soul Advisor 架構調整 — 可行性分析與建議

## 一、目標架構摘要

您提出的新模式重點：

| 維度 | 現行 | 新架構（Soul Advisor） |
|------|------|------------------------|
| **人設** | 伯彥老師（專業命理分析師） | 一對一諮商式指導者（Soul Advisor），引導靈魂閱讀「人生遊戲任務書」 |
| **語氣** | 肯定清晰、不宿命不恐嚇 | 溫暖、深度、諮商式；描述在前、技術在後；不得恐嚇／宿命論／絕對語氣 |
| **書結構** | 20 章（s01–s20） | 0–12 主章 + 子節（2-1～2-4、4-1～4-4） |
| **標題** | 技術名（如「命宮綜合分析」） | 人話描述（技術名稱），例：紅燈高壓區（子女宮） |
| **單章輸出** | 四欄：structure_analysis, behavior_pattern, blind_spots, strategic_advice | 固定模板：① 故事式導入 ② 行為模式 ③ 盲點 ④ 建議 ⑤【技術標記】 |
| **星曜文案** | 由系統傳入星曜宮位評語，模型延伸 | 「由 Cursor DB 自動帶入」主星／輔星 feature 插入位 |

## 二、可行性結論：**可行，建議分階段**

整體技術上可行，但涉及：**章節對應、輸出格式、模板遵守率、前端顯示**。建議分階段實作以控制風險並保留回滾空間。

## 四、輸出格式：四欄 vs 五段＋技術標記

**現行**：  
每章 JSON 固定四欄  
`structure_analysis` / `behavior_pattern` / `blind_spots` / `strategic_advice`  
→ Worker 驗證、KV 儲存、Viewer（SectionCard）皆依此渲染。

**新設計**：  
① 故事式導入 ② 行為模式 ③ 盲點 ④ 建議 ⑤【技術標記】

**建議（相容現有 Viewer）**：

- **方案 A（最小改動）**  
  - 仍輸出四欄。  
  - 在 prompt 中規定：**人話放前、技術放後**；`structure_analysis` = ① 故事＋段落末 ⑤ 技術標記（同一欄內，技術集中最後）；`behavior_pattern` = ②；`blind_spots` = ③；`strategic_advice` = ④。  
  - 優點：無需改 Worker/Viewer/assembler。  
  - 缺點：技術與故事在同一欄，若未來要分開摺疊需再拆欄。

- **方案 B（推薦中長期）**  
  - 新增第五欄 `technical_marker`（可選）。  
  - 規定：①→structure_analysis，②→behavior_pattern，③→blind_spots，④→strategic_advice，⑤→technical_marker。  
  - Worker：驗證與儲存改為 4+1 欄（無 technical_marker 時可為空字串）。  
  - Viewer：SectionCard 在四塊之後，若有 `technical_marker` 再渲染「技術依據」摺疊區。  
  - 優點：版型完全符合「描述在前、技術在後」且可單獨摺疊；assembler 改動有限。

**建議**：先以 **方案 A** 只改 prompt 上線，確認語氣與結構穩定後，再導入 **方案 B** 的第五欄與 UI。

## 五、固定模板與模型遵守度

新架構有大量固定模板（宮位六段式、五行／大限／流年／小限／靈魂使命／業力／三條短期長期避凶／三盤疊加／總結）。  

**可行性**：  
- 可行：在 system / 每題 template 中**明確寫出**「必須依下列 ①～⑤ 順序輸出，且 ⑤ 為【技術標記】」並給範例，多數模型能大部分遵守。  
- 風險：章節一多、模板一長，偶爾會漏段或順序錯，需在 **post-process** 或 **retry** 時做簡單檢查（例如是否含「技術標記」、段落數是否合理）。

**建議**：  
- 每題的 `template` 在 worker 內改為您提供的對應模板（六段式／五行／大限／流年／小限／靈魂使命／業力／三條／三盤疊加／總結）。  
- 在 system 或題目開頭加一句：「本題必須嚴格依下方 ①～⑤（或 ①～④＋技術標記）順序撰寫，不可跳段。」

## 六、「Cursor DB 自動帶入」星曜 feature

您提到「星曜 feature 由 Cursor DB 自動帶入」。  

**解讀與建議**：

- 若指**現有「星曜宮位評語」**（expert-admin 後台／KV 或 API 傳入的 `starPalaces`）：  
  - 目前已在 infer / generate-section 的 user 中注入，並要求模型「延伸說明、勿重複評語」。  
  - 只要在 prompt 中改寫為：「【主星與輔星 feature 插入位】由系統自動帶入，請在段落最上方引用後，再寫 ① 故事…⑤ 技術標記」，即可對齊新格式，**無需新 DB**。

- 若指**另建一個 Cursor / 內容庫**，專門存「天相：協調、穩定」「七殺：行動力、突破力」等制式句：  
  - 可行：在 Worker 或內容服務中，依命盤星曜列表查詢該庫，組成一塊「星曜 feature 插入位」文字注入 prompt。  
  - 需新增：內容來源（例如 D1 表、KV、或靜態 JSON）與查詢邏輯。

**建議**：  
- 第一階段先用現有 `starPalaces` 當作「feature 插入位」來源，僅改寫 prompt 說明。  
- 若日後要統一多本命書的星曜說明，再建「星曜 feature DB」並在組裝 prompt 時帶入。

## 七、標題格式「人話描述（技術名稱）」與宮位紅綠燈

- **標題**：  
  - 目前 section 的 `title` 來自 `SECTION_TEMPLATES`（如「命宮綜合分析」）。  
  - 改為「人話描述（技術名稱）」時，可：  
    - 在模板中為每題訂好新 title（例如「你是帶著什麼狀態上場？（命宮）」），或  
    - 由模型在該題輸出中建議標題，再由系統寫入（需多一個欄位或從第一段擷取）。  
  - **建議**：在 `lifeBookTemplates.ts` 直接改每題的 `title` 為新格式，最簡單且穩定。

- **宮位紅綠燈標題**（🚫紅燈高壓區（子女宮）等）：  
  - 需知道「哪一宮是紅／黃／綠」。  
  - 現有 `weight_analysis.traffic_signals` 或宮位權重可支援。  
  - **建議**：在組裝 s05–s14（及 s20）的 user 時，依 traffic 或權重寫入「本題宮位對應燈號」，並在 template 中規定標題格式為「🚫紅燈高壓區（OO宮）」等，由模型或由系統依資料直接生成標題字串。

## 八、實作建議（分階段）

### Phase 1：人設＋語氣＋描述在前技術在後（僅改 prompt，不改章節數與輸出欄位）

- 將 **INFER_SYSTEM_PROMPT** 與 **getSystemPrompt()** 改為 Soul Advisor 人設與語氣規範（溫暖、諮商式、不恐嚇不宿命、描述在前技術在後）。  
- 各題 **template** 維持現有 20 題結構，但在說明中加入「① 故事 ② 行為 ③ 盲點 ④ 建議，技術集中於 structure_analysis 段落末或最後一段」。  
- **不新增** 0／12，**不**改 SECTION_ORDER 或 Viewer。  
- **產出**：語氣與編排已貼近新架構，可快速驗證效果與模型表現。

### Phase 2：章節重排與標題改為「人話（技術名）」

- 調整 **SECTION_ORDER** 與 **SECTION_TEMPLATES**，對齊新書目 0–12（含 2-1～2-4、4-1～4-4 對應到現有 s03/s02/s10/s01、s15/s16 等）。  
- 新增 **s00**（靈魂行前簡報）、**s21 或等同**（靈魂總結），並在 infer + generate-section 加上對應題。  
- 將每題 **title** 改為您提供的「人話描述（技術名稱）」。  
- 宮位題（3）的標題：依 traffic／權重注入「本題宮位＝紅/黃/綠」，並在 template 規定標題格式。  
- 仍維持四欄輸出，必要時在 prompt 中明確「⑤ 技術標記寫在 structure_analysis 最後一段」。

### Phase 3：固定模板＋五段式與 optional 第五欄

- 每題 **template** 改為您提供的完整固定模板（六段式宮位、五行、大限、流年、小限、靈魂使命、業力、三條、三盤疊加、總結）。  
- 若採用 **方案 B**：  
  - 在 Worker 的 generate-section 解析與儲存中支援 `technical_marker`（第五欄）；  
  - Viewer SectionCard 與 assembler 支援顯示／摺疊「技術依據」區塊。  
- 視需要加入簡單 **post-check**（是否含技術標記、段落數）與 retry 邏輯。

### Phase 4（可選）：星曜 feature 庫與 0／12 專用邏輯

- 若有「星曜 feature 庫」需求：建 D1/KV 或 JSON，在組裝 prompt 時依命盤星曜帶入「主星與輔星 feature 插入位」。  
- 為章 0、章 12 設計專用 infer 子步驟或單獨題目，確保有總覽與收束。

## 九、風險與注意

| 風險 | 緩解 |
|------|------|
| 模型偶爾不照 ①～⑤ 順序或漏技術標記 | 在 template 開頭強制寫明「必須依 ①～⑤ 順序」；必要時 post-check + 重試 |
| 新章節 0／12 增加 token 與呼叫次數 | 先以 Phase 1 驗證再擴充；0 可考慮用 infer 總覽壓成一段 |
| 宮位紅綠燈標題依賴權重／traffic 正確性 | 後端確保 decadalLimits / traffic_signals 正確注入；若缺則該宮用「重要調整區」等中性標題 |
| Viewer 需支援新標題或第五欄 | Phase 2 只改 title 字串；Phase 3 再改 SectionCard／assembler，並保持向後相容（無 technical_marker 時不顯示） |

## 十、總結

- **可行性**：可行；新架構與現行 s01–s20 多數可對應，僅需新增 0、12 與標題／呈現邏輯。  
- **輸出格式**：建議先維持四欄、在 structure_analysis 內「描述在前、技術在後」（方案 A），再視需要引入第五欄 `technical_marker`（方案 B）與 Viewer 摺疊。  
- **星曜 feature**：可先用現有 starPalaces 當「插入位」，再決定是否建專用 DB。  
- **實作**：分四階段（人設與語氣 → 章節與標題 → 固定模板與五段式／第五欄 → 星曜庫與 0／12 專用），可控制風險並保留回滾空間。

若您希望先從某一 Phase 開始實作，我可以依選定階段給出對應的 prompt 片段與檔案修改清單（lifeBookInfer.ts、lifeBookTemplates.ts、lifeBookPrompts.ts、SectionCard/assembler 等）。
